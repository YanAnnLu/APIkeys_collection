# UI/UX 開發契約

最後更新：2026-05-24

這份文件用來避免 UI/UX 需求在聊天中漂移。使用者可以拿 Foxy、Steam、tem 或其他軟體當作例子，但那些例子只代表「互動精神」或「心流參照」，不代表本專案要沿用它們的命名、資訊架構或商業語彙。

## 核心規則

1. **參照不是命名**
   - 例如「像 Foxy」只表示清單雙擊、下載隊列、續傳心流等操作精神。
   - 除非使用者明確要求，程式碼、文件、UI 文案不得把參照軟體名稱寫成正式概念。

2. **先定義 UX 契約，再寫 UI**
   - 中大型 UI 變更不能直接從口頭比喻開寫。
   - 先寫出：操作對象、觸發方式、狀態變化、後端服務、錯誤狀態、驗收方式。

3. **Tk / Qt 只是皮**
   - 爬蟲、下載、界域表單、設定檔、健康狀態都先落在後端契約或 view-model。
   - Tk 只接上既有契約；未來 Qt 可以重用同一組服務。

4. **快速交付可以粗，但不能假**
   - 可以先用短路徑、fixture、有限資料量或本機下載資料夾。
   - 不可以做假的百分比、假的下載、假的成功狀態，或只有裝飾作用的按鈕。

5. **避免巨型 UI 檔案**
   - 新 UI 能力應拆成小 dialog、小 workflow、小 view-model 或後端 service。
   - 不把臨時展示功能直接塞回 `launcher_ui.py` 形成新的巨石。

## UI 需求進入實作前，必須回答

每個 UI 功能在動工前至少要能回答：

| 問題 | 說明 |
| --- | --- |
| 使用者現在想操作哪個物件？ | 例如 crawler asset、dataset seed、download plan item、local install。 |
| 入口在哪裡？ | 分頁、卡片、清單、右鍵、齒輪、工具列或對話框。 |
| 使用者觸發了什麼？ | 單擊、雙擊、按鈕、切換開關、拖放、選擇資料夾。 |
| 後端服務是哪一個？ | 例如 `crawler_asset_service`、`crawler_asset_profiles`、`ingestion_pipeline`。 |
| 狀態怎麼變？ | idle、loading、paused、running、failed、archived、needs_credentials。 |
| 失敗時怎麼說？ | 要有可行的下一步，不只顯示 exception。 |
| 怎麼驗收？ | 單元測試、py_compile、GUI smoke、真實下載檔案、事件 log 或 CI。 |

如果答不出來，就先寫 UX 契約或 OpenSpec，不直接改 UI。

## 爬蟲資產目前的互動契約

### 資產分頁

- 第一分頁應是「爬蟲資產」：管理入口、爬蟲狀態、設定、健康度、Logo、啟用/封存。
- 第二分頁才是「下載器」：管理下載清單、隊列、進度、暫停/續傳、輸出資料夾。

### 爬蟲卡片

一個 crawler asset 應該像一張清爽卡片：

- Logo 或自定義圖片。
- 名稱與來源說明。
- 健康狀態 emoji / badge。
- seed / dataset / capability 摘要。
- 啟用/封存狀態。
- 明確的齒輪或「爬蟲設定」按鈕。

卡片上的設定入口必須是顯式控制，不用雙擊暗示。

### 雙擊語意

雙擊保留給「清單項目啟動」：

- 在下載器清單中雙擊某一列，表示把該項目送進下載隊列或啟動選中的下載項。
- 爬蟲卡片的設定不使用雙擊，以免和下載器心流衝突。

### 爬蟲三能力

每個入口原則上由一個 crawler asset 封裝，並具備三個後端能力槽：

1. `fetch_metadata()`：取得入口/資料集基本 metadata。
2. `list_datasets(bounds?)`：列出 seed、資料集或版本候選。
3. `build_download_plan(dataset, bounds)`：依使用者界域產生下載計畫。

帳號、API key、限流、登入需求、terms risk 屬於 crawler asset profile，不屬於 dataset 本身。

### 界域表單

界域不是固定欄位，必須由資料結構推導：

1. 先 probe metadata 或 head sample。
2. 得到欄位、時間欄、空間欄、版本欄、格式、限制條件。
3. 轉成 `bounds_schema`。
4. UI 根據 `bounds_schema` 動態生成表單。
5. 使用者輸入界域後，交給 `build_download_plan()`。

Tk 版只需要先能呈現基本動態表單；Qt 版未來可以換成更漂亮的 skin。

## 開發模式切換

### 日常開發模式

- 優先後端契約、測試、防禦性錯誤處理、文件同步。
- UI 變更先小片切開，避免巨型重寫。
- 中大型變更要補 OpenSpec 或 GTD 條目。

### 快速交付 / 展示模式

- 先做能被人操作的穩定 GUI 路徑。
- 進度、下載、錯誤都要是真的。
- 可以降低顆粒度，但要留下可回收回主線的服務邊界。
- 展示結束後，要把壓力測試學到的教訓回收進 GTD、handoff、OpenSpec 或本文件。

## 實作驗收

UI/UX 變更完成時，回報至少包含：

- 本輪改了哪個心流。
- 哪些參照只被當作精神，沒有變成命名。
- 後端服務邊界在哪裡。
- 使用者可以如何操作。
- 測試與 CI 狀態。
- 剩餘 UX 風險與下一輪優化方向。
