# RRKAL 展示壓力測試回顧

## 這次學到的事

這次展示急改證明：展示模式不是一次性假功能，而是正式產品在高壓情境下的最短可操作切片。它必須能讓人類操作者不改程式碼，只靠 GUI 完成穩定流程。

## 防禦性編程與快速交付的分流

- 防禦性編程：用完整測試、清楚邊界、可維護模組與文件，穩健實現產品能力。
- 快速交付：用最短安全 GUI 路徑重現已存在或接近完成的能力，不能偽造進度、不能假下載、不能只靠 CLI。
- 展示後回收：快速交付產物要轉回正式架構，補上測試、錯誤處理、文件與 OpenSpec/GTD 任務。

## 本次具體教訓

- GUI 啟動本身要成為 smoke gate；不能只相信後端測試通過。
- 進度條必須來自真實下載事件或真實階段狀態，不能用裝飾性百分比。
- 展示資料夾要可追蹤，但本機依賴、log、SQLite 與驗證輸出仍應忽略。
- 地端測試要優先使用本機 C 槽 clone，雲端碟可保留為開發工作區，但不應作為展示穩定性的唯一證據。
- 急改前要先區分「可展示穩定路徑」與「仍在實驗的功能」，避免把半成品暴露成正式功能。

## 後續回收方向

- 將展示下載流程收斂為正式 GUI library action。
- 將 seed 覆蓋報告變成可重跑的正式狀態面板。
- 將本次 GUI 啟動問題納入 pre-push 或展示前 smoke。
- 為展示模式保留可控樣本大小、資料夾選擇、fallback 來源說明與可機器讀取 summary。
